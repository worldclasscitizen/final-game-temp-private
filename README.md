# 죽을수록 강해진다 — Phase 2 Prototype

Nidhogg + ROUNDS 하이브리드 게임의 Godot 4 프로토타입입니다.
현재 단계(Phase 2)는 **코어 루프 + 카드 시스템** 을 포함합니다.
온라인 멀티플레이와 로컬 2P 두 모드 모두 지원합니다.

## 이 프로토타입에 포함된 것

- [x] 가로로 긴 2D 맵 (2400×720)
- [x] 이동/점프/조준/발사
- [x] 원샷 원킬 총알 → 사망 → 1초 뒤 자동 리스폰
- [x] 상대 진영 끝 도달 시 승리 판정
- [x] **온라인 멀티플레이** (Godot ENetMultiplayerPeer)
- [x] **로컬 2P** — 같은 PC 에서 두 명이 플레이 (네트워크 불필요)
- [x] **카드 시스템** — 사망 시 3장 중 하나를 골라 총알 능력치 업
- [ ] 카메라 이탈 자동 사망 (Phase 3 — 현재는 단순 중앙 카메라)
- [ ] 픽셀 아트/사운드 (Phase 4)

## 카드 목록 (Phase 2)

사망할 때마다 3장이 랜덤으로 뽑히고, 중복 획득이 가능합니다.

| 카드 | 효과 |
|---|---|
| **쾌속** | 탄속 +35% (중첩 가능) |
| **대구경** | 탄환 크기/히트박스 +50% (중첩 가능) |
| **유도** | 가장 가까운 적을 향해 곡선 주행 (회전 속도 증가) |
| **반사** | 벽에 맞으면 튕김 (+1 회) |
| **관통** | 벽을 뚫고 지나감 (+1 회) |

낙사(y>2000) 로 인한 사망은 카드를 주지 않습니다. 정직하게 총에 맞아야 강해집니다.

## 기술 스택

- Godot **4.3** 이상 (Forward Plus 가 아닌 **GL Compatibility** 렌더러로 설정됨 — 어디서든 잘 돌아가도록)
- GDScript
- 네트워킹: Godot 내장 `ENetMultiplayerPeer` + `MultiplayerSynchronizer` + `rpc()`

## 실행 방법

1. [Godot 4.3 이상](https://godotengine.org) 설치
2. Godot 에디터에서 "Import" → 이 폴더의 `project.godot` 선택
3. F5 로 실행 → 메인 메뉴가 뜹니다 (호스트 / 접속 / 로컬 2P)

### 로컬 2P 모드 (같은 PC · 제일 빠른 테스트)

1. 메인 메뉴 → **로컬 2P (같은 PC)** 버튼 클릭
2. 창 하나에서 두 명이 동시에 조작 (조작키는 아래 표 참고)
3. ESC 또는 "메인으로" 버튼으로 빠져나옵니다

### 온라인 멀티 — 같은 PC 에서 두 개 창으로 테스트

1. Godot 에디터 → Project Settings → Debug → `Run Multiple Instances` 를 **2** 로 설정
2. F5 로 실행 → 두 개의 창이 동시에 뜸
3. 창 A 에서 "호스트 시작"
4. 창 B 에서 IP 란 비워두고 (= 127.0.0.1) "접속"
5. 자동으로 게임 씬 전환

### 온라인 멀티 — 다른 PC 와 플레이

1. 한 명이 "호스트 시작" — 방화벽에서 9999 포트 허용 필요
2. 같은 네트워크(LAN) 라면 호스트의 로컬 IP 를 입력해서 접속
3. 인터넷 너머로 하려면 호스트가 포트포워딩 또는 ngrok/ZeroTier 같은 툴 필요

## 네트워크 설계 (복습)

Spring Boot 백엔드 경험이 있으시니 설명이 덜 필요할 것 같지만, Godot 식 관점으로:

**1. 권한(Authority) 모델**
- 각 플레이어 캐릭터의 `multiplayer_authority` 는 해당 peer_id 로 설정
- 입력/이동은 `is_multiplayer_authority()` 가 true 일 때만 처리
- 즉 "내 캐릭터는 내가 움직이고, 상대 캐릭터는 네트워크로 받은 값만 반영"
- 로컬 모드에서는 네트워크가 없으므로 `input_scheme` 으로 조작 가능 여부를 판정 (`player.is_controllable()`)

**2. 상태 동기화**
- `MultiplayerSynchronizer` 가 position / aim_angle / facing / is_alive 를 주기적으로 동기화
- replication_mode: position, aim_angle 은 `ALWAYS`(매 틱), facing, is_alive 는 `ON_CHANGE`(값 바뀔 때만)
- 로컬 모드에서는 Synchronizer 가 활성 상태여도 네트워크 피어가 없으므로 사실상 no-op

**3. 이벤트 전파**
- 발사: `_fire_bullet.rpc(pos, angle, shooter_id, card_ids)` — 모든 피어가 같은 초기값과 카드 조합으로 로컬 총알 스폰
- 사망: **피격 당한 플레이어가 자기 권한으로 `_apply_die.rpc()` 선언** — 이중 사망/슈터 측 거짓 판정 방지
- 카드 획득: `_apply_card.rpc(card_id)` — 양쪽 피어 모두 같은 카드 배열을 갖도록
- 승리: `_announce_win.rpc(winner_id)`
- 로컬 모드에서는 모든 `.rpc()` 대신 직접 함수 호출로 분기

**4. 카드 효과의 결정론성**
- 카드는 `Array[String]` (id 리스트) 로 관리 → 모든 피어가 같은 스탯 계산 결과를 얻음
- 발사 시점에 `cards.duplicate()` 로 스냅샷을 전송 → 두 피어의 총알 초기 스탯이 일치
- 유도 탄환은 `_find_target()` 으로 가장 가까운 적을 찾는데, 양쪽의 플레이어 위치가 동일하므로 같은 타겟을 같은 프레임에 선택

## 알려진 제약 (Phase 2)

- **카메라가 단순**: 두 플레이어의 중앙만 따라감. 제안서의 "이기는 쪽을 따라가고 뒤진 쪽은 자동 사망" 은 Phase 3 에서 구현
- **낙사 리스폰 시 카메라 밖 자동사망 개념 없음**: 맵 y > 2000 이면 죽습니다 (임시 가드)
- **지연 보정 없음**: 핑 200ms 이상이면 피격 판정 체감 이상해질 수 있음. LAN 에서 테스트 권장
- **유도 탄환의 미세한 양쪽 차이 가능성**: 두 피어의 타임스텝 drift 가 쌓이면 유도 궤적이 조금씩 갈릴 수 있음. 결정적 치명도는 낮지만, Phase 3 에서 탄환 위치도 간헐적으로 sync 하는 것을 고려
- **카드 풀이 5종**: 로드맵 상 Phase 2 최소 집합. 가감/멀티샷/체력/이속 등은 다음 스프린트에서 추가 예정

## 트러블슈팅

**"포트 9999 를 열 수 없습니다"**
- 이미 다른 프로세스가 쓰고 있음. `netstat -ano | findstr 9999` 로 확인하고 종료

**로컬 2P 에서 P2 가 카드를 못 고름**
- P2 전용 카드 키는 7/8/9. 1/2/3 은 P1 용이라 P2 창이 따로 뜨면 아래쪽에 위치
- 마우스 클릭으로도 P2 창을 누를 수 없게 막혀있음 (P1 이 자기 클릭으로 P2 카드를 고를 수 있는 버그 방지)

**한쪽에서만 상대가 안 보임 (온라인)**
- 두 피어의 게임 씬 로드 타이밍 차이 가능. 일단 메인으로 돌아갔다가 재연결
- 영구적으로 재현되면 `player.gd` 의 `_enter_tree()` 권한 설정 부분 로그 찍어보기

**총알에 맞았는데 안 죽음**
- 피격자 측에서 판정하는 구조라, 본인 창이 렉 걸린 순간이라면 충돌이 먼저 해소될 수 있음
- Phase 3 에서 tolerant hit (서버 측 되돌아보기) 를 넣을 예정
